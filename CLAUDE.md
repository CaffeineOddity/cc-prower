# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Build
npm run build          # Compile TypeScript, copy hooks/templates to dist/

# Development
npm run dev            # Start with tsx (for debugging)
npm start              # Start built CLI

# Testing & Quality
npm test               # Run vitest tests
npm run lint           # Run ESLint
npm run typecheck      # TypeScript type checking

# Important: Test files must be placed in e2e-test/ directory, not cc-power-service/
# Always run tests after making code changes

# CLI (after build)
ccpower start          # Start the service
ccpower run <path>     # Run a project with Tmux integration
ccpower init <name> --provider <type>  # Initialize a new project
ccpower logs <name>    # View message logs
ccpower status         # Show service status
```

## Architecture Overview

CC-Power is a lightweight bridge connecting Claude Code with chat platforms (Feishu, Telegram, WhatsApp). The service uses MCP (Model Context Protocol) via stdio transport only.

### Core Flow

1. **Incoming**: Chat platform → Provider → Router → MessageInjector → tmux → Claude Code
2. **Outgoing**: Claude Code → stop-hook → Router → Provider → Chat platform

### Key Components

**Router** (`core/router.ts`)
- Central message routing coordinator
- Manages: ProviderRegistry, MessageQueue, MessageInjector, TmuxSessionPoller, MessageLogger
- Maps `chatId → MessageRoute` for response routing

**ProviderRegistry** (`core/provider-registry.ts`)
- Manages provider lifecycle (register/unregister)
- Tracks tmux panes by project name
- Returns provider instances by projectId

**MessageInjector** (`core/message-injector.ts`)
- Injects messages into tmux sessions using `tmux send-keys`
- Safety check: verifies current process is safe (claude/node/bash/zsh) before injection
- Unregisters project if tmux session dies

**TmuxSessionPoller** (`core/tmux-session-poller.ts`)
- Polls tmux sessions for health
- Auto-unregisters dead projects via callback

**TmuxExecutor** (`core/tmux-executor.ts`)
- Low-level tmux command execution
- Methods: `listPanes()`, `sendKeys()`, `hasSession()`, `getSessionName()`

**BaseProvider** (`providers/base.ts`)
- Abstract base class for all providers
- Methods to implement: `connect()`, `sendMessage()`
- Message callback via `onMessage()`
- `getProjectId()` uses `get_project_id()` function

### Provider Pattern

All providers extend `BaseProvider`:
- `FeishuProvider` - uses @larksuiteoapi/node-sdk
- `TelegramProvider` - uses node-telegram-bot-api
- `WhatsAppProvider` - Business API integration
- Custom Provider (in development) - WebSocket-based

Each provider:
1. Implements `connect()` to establish platform connection
2. Calls `emitMessage()` for incoming messages
3. Implements `sendMessage()` for outgoing responses

### Signal-Based Registration

Projects auto-register via file system signals in `~/.cc-power/signals/`:

- `register-{project_name}.json` - Created by `ccpower run`
- `unregister-{project_name}.json` - Created on cleanup

Signal format (`utils/signals.ts`):
```typescript
interface RegisterSignal {
  type: 'register';
  projectId: string;        // Generated by get_project_id()
  projectName: string;
  tmuxPane: string;         // Format: "session:0"
  projectDirectory: string;
  config: TemplateProviderConfig;
}
```

### Project ID Generation

`get_project_id()` in `types/provider.config.ts`:
- Feishu: `{app_id}-{chat_id}`
- Telegram: `{bot_token}-{chat_id}`
- WhatsApp: `{phone_number}-{chat_id}`
- Custom: `{app_id}-{project_name}`

### Tmux Integration

- Session name = project_name (sanitized)
- Pane format = `session:0.0`
- Commands use pane targeting: `tmux send-keys -t {pane} {text}`
- `ccpower run` creates shared sessions for local + remote users

### Message Types

```typescript
// Incoming message
interface IncomingMessage {
  type: 'incoming';
  provider: string;
  projectId: string;
  chatId: string;
  userId: string;
  userName?: string;
  content: string;
  metadata?: Record<string, any>;
}

// Outgoing message
interface OutgoingMessage {
  type: 'outgoing';
  provider: string;
  projectId: string;
  chatId: string;
  content: string;
  metadata?: Record<string, any>;
}
```

### Configuration

- Global config: `~/.cc-power/config.yaml` (or `config.yaml` in project root)
- Project config: `.cc-power.yaml` in project directory
- Provider templates: `providers/templates/{feishu,telegram,whatsapp}-template.yaml`

Global config structure:
```yaml
WebSocket:
  port: 8080
  host: "127.0.0.1"

logging:
  level: debug|info|warn|error
  file: optional_path

providers:
  feishu: { enabled: true }
  telegram: { enabled: true }
  whatsapp: { enabled: true }
  internal: { enabled: true }  # For Custom Provider
```

### Type Definitions

Key types in `types/`:
- `index.ts` - Exports IProvider, IRouter, ILogger
- `message.ts` - IncomingMessage, OutgoingMessage, MessageRoute
- `provider.config.ts` - Provider-specific configs, get_project_id()
- `ccpower.config.ts` - GlobalConfig, ProjectConfig, WebSocketConfig

### Adding a New Provider

1. Create `providers/{name}.ts` extending `BaseProvider`
2. Implement `connect()` and `sendMessage()`
3. Add to `ProviderType` in `types/provider.config.ts`
4. Add `get_project_id()` case for new provider
5. Create template in `providers/templates/{name}-template.yaml`
6. Import and register in `core/provider-registry.ts`
